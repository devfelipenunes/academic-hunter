MCP Tools Reference

Academic Hunter exposes 15 tools via the Model Context Protocol (MCP). Any MCP-compatible client — Claude Desktop, Cursor, or custom agents — can drive the research pipeline autonomously.

Setup

Add the following server configuration to your MCP client. The auto-installer (python install.py) can do this for you. 

{
  "mcpServers": {
    "academic-hunter": {
      "command": "/path/to/academic-hunter/venv/bin/python",
      "args": ["-m", "academic_hunter.interfaces.mcp.server"],
      "env": {
        "PYTHONPATH": "/path/to/academic-hunter/src"
      }
    }
  }
}

After connecting, the tools appear automatically in your MCP client. Use the quick_topic_discovery tool first to identify jargon, then configure your search via update_config, and run it with run_search.

Configuration

4 tools

read_config

Configuration

Returns the current full search configuration from config.json. Use this to check which anchors or technical strings are currently configured before planning a search.

read_config() → str

Returns: JSON string of the current configuration (settings, anchors, technical strings, weights, context rules).

# Example — ask your MCP client:
"Read my current Academic Hunter config."

# Result:
{
  "settings": {
    "min_relevance_score": 3.5,
    "start_year": 2021,
    "limit_per_query": 100
  },
  "anchors": {
    "Blockchain_Projects": ["CBDC", "digital currency"]
  },
  ...
}

update_config

Configuration

Updates the project's config.json with new research parameters. A backup is automatically saved to the SQLite history before applying changes. You can provide a full or partial SearchConfigUpdate object.

update_config(config_update: SearchConfigUpdate) → str
Parameter Type Required Default Description
topic Optional[str] No — Short name for this config (e.g., "CBDC Latency 2025")
research_strategy Optional[str] No — Paragraph describing your research plan and taxonomy pillars
settings Optional[Dict] No — System settings (min_relevance_score, start_year, limit_per_query)
anchors Optional[Dict] No — Mandatory keyword groups. At least one term from each group must match.
technical_strings Optional[Dict] No — Taxonomy categories for your technical terms
technical_weights Optional[Dict] No — Weighted terms (5.0=core niche, 3.0=protocols, 1.5=generic)
context_rules Optional[Dict[str, List[str]]] No — Disambiguation rules — maps ambiguous terms to required context words
keyword_only_terms Optional[List[str]] No — Generic keywords for fallback search
keyword_only_category Optional[str] No — Aggregating category for keyword-only terms

Returns: Confirmation message with the applied configuration summary.

# Example — ask your MCP client:
"Configure Academic Hunter for CBDC latency research starting from 2022."

# The agent calls update_config with appropriate anchors and weights.
# Result:
"Configuration updated successfully. Backup saved to history (ID: 7)."

list_config_history

Configuration

Lists previous configurations saved in the MCP SQLite database. Useful for finding a config ID to restore with restore_config_by_id.

list_config_history(limit: int = 5) → str
Parameter Type Required Default Description
limit int No 5 Number of history entries to return

Returns: Formatted list of config entries with ID, timestamp, and topic.

# Example — ask your MCP client:
"Show me my recent config changes."

# Result:
ID: 7 | 2026-07-15 14:32:01 | CBDC Latency Research
ID: 6 | 2026-07-14 09:15:44 | Blockchain Governance
ID: 5 | 2026-07-12 18:00:22 | DeFi Cross-Chain

restore_config_by_id

Configuration

Restores a previous configuration backup by its database ID. Use list_config_history first to find the desired config ID.

restore_config_by_id(config_id: int) → str
Parameter Type Required Default Description
config_id int Yes — ID of the configuration to restore (from list_config_history)

Returns: Confirmation message with the restored configuration summary.

# Example — ask your MCP client:
"Restore config ID 5 from my history."

# Result:
"Configuration restored from ID 5 (DeFi Cross-Chain, saved 2026-07-12)."

Search / Execution

2 tools

run_search

Search

Executes the full academic search pipeline. Spawns concurrent workers for each API connector (Semantic Scholar, OpenAlex, Crossref, ArXiv, CORE, DBLP, DOAJ), deduplicates results, scores them, generates reports, and indexes papers into the vector store.

run_search(limit_per_source: int | None = None) → str
Parameter Type Required Default Description
limit_per_source int | None No None Override the global limit_per_query from config.json for this run

Returns: Path to the generated Markdown report and summary of results.

# Example — ask your MCP client:
"Run a search for the current configuration."

# Or with a custom limit:
"Run a quick search with just 10 results per source."

# Result:
"Search complete. 47 papers found, 23 passed relevance scoring.
Report saved to results/FLUXO_PRISMA_20260715_103022.md"

Discovery / Citation Graph

4 tools

explore_citation_graph

Discovery

Explores the citation graph of a paper using its DOI via Semantic Scholar. Direction can be "citations" (papers that cited this DOI) or "references" (papers this DOI cited). Useful for snowballing — finding relevant papers through citation chains.

explore_citation_graph(doi: str, direction: str = "citations") → str
Parameter Type Required Default Description
doi str Yes — DOI of the paper to explore
direction str No "citations" "citations" (who cited this) or "references" (what this cited)

Returns: Formatted list of papers with titles, authors, and years.

# Example — ask your MCP client:
"Find papers that cited doi:10.1109/ACCESS.2023.3267890"

# Or find references:
"What are the references of doi:10.1016/j.future.2023.01.005?"

# Result:
"CITING PAPERS:
1. 'CBDC Architectures' — Smith et al. (2024)
2. 'Digital Euro Design' — Garcia et al. (2023)"

Export

1 tool

export_to_obsidian

Export

Exports a formatted Markdown note directly to your Obsidian vault. The vault path is configured in config.json under settings.obsidian_vault_path. Generates proper YAML frontmatter with title, date, and tags.

export_to_obsidian(topic: str, content: str, tags: list | None = None) → str
Parameter Type Required Default Description
topic str Yes — Title/topic of the note (used in frontmatter and filename)
content str Yes — Markdown content of the report
tags list | None No None Optional list of Obsidian tags (e.g., ["research", "CBDC"])

Returns: Confirmation with the vault path where the note was saved.

# Example — ask your MCP client:
"Export this research summary to my Obsidian vault with tags
'CBDC, latency, blockchain'."

# Result:
"Note exported successfully to:
Obsidian/Academic_Hunter/CBDC_Latency_Review.md"

RAG / Vector Store

4 tools

semantic_search

RAG

Performs semantic (embedding-based) search across all previously indexed papers using ChromaDB. Finds conceptually related papers even when exact keywords don't match. Uses ONNX-based all-MiniLM-L6-v2 embeddings.

semantic_search(query: str, top_k: int = 10, score_threshold: float = 0.0) → str
Parameter Type Required Default Description
query str Yes — Natural language query
top_k int No 10 Maximum number of results (max 50)
score_threshold float No 0.0 Minimum relevance threshold (0.0 to 1.0)

Returns: Formatted results with relevance scores, metadata, and abstract previews.

# Example — ask your MCP client:
"Search my indexed papers for 'cross-chain atomic settlement'"

# Result:
"1. 'Atomic Settlement Mechanisms' (2024) — Relevance: 89%
   DOI: 10.1234/atomic — Venue: IEEE Access
   Abstract: This paper proposes a novel atomic settlement...

2. 'DvP Protocols' (2023) — Relevance: 76%
   DOI: 10.5678/dvp — Venue: Future Generation CS
   Abstract: Delivery-versus-payment mechanisms..."

index_papers

RAG

Indexes the latest search results into the ChromaDB vector store. This is also called automatically at the end of each pipeline run, but can be invoked manually to re-index or index from previous runs.

index_papers() → str

Returns: Confirmation with the number of papers indexed and collection name.

# Example — ask your MCP client:
"Index the latest papers for semantic search."

# Result:
"Indexing complete. 23 papers indexed to collection 'papers'."

RAG

Shows statistics about the vector store: available collections, paper count, and status. Use this to check if papers have been indexed and how many are available for semantic search.

vector_store_stats() → str

Returns: Collection name, total paper count, and status.

# Example — ask your MCP client:

# Result:
"Collection: papers | Total papers: 47 | Status: Ready"

Answers a research question by retrieving relevant papers from the vector store and presenting their content as context. Designed for AI-powered answers grounded in your actual paper collection.
